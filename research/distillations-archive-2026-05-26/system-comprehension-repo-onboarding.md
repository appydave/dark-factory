# Unified Skill: Repo Onboarding (system-context upgrade)

**Purpose**: One file, `CONTEXT.md`, that any agent reads once to understand any codebase — its purpose, abstractions, workflows, design decisions, constraints, expert mental model, scope limits, and failure modes. Generated by crawling source files; refreshed on staleness signals.

## Intent

Generate or refresh a persistent CONTEXT.md document that eliminates repeated codebase crawls by encoding all 8 required dimensions of system comprehension into a single shareable artifact.

## Winner's Mechanism

`appydave-plugins:skill:system-context` is the winner. It is David's own production skill, already installed and triggered daily. It specifies:

- Explicit **8-dimension checklist** (purpose / core abstractions / key workflows / design decisions / non-obvious constraints / expert mental model / scope limits / failure modes) — no other artifact in the cluster uses a named multi-dimension checklist
- **Two-folder convention** (`context/` committed vs `.context/` machine-local) — the cleanest design decision in the cluster; solves the "should context ship with the repo?" question with a folder name, no flags
- **Refresh vs first-run detection** — preserves manually-added annotations; re-reads sources for staleness
- **context.globs.json** co-generation — downstream skills (`app-query`, `llm-context`) depend on it; no other cluster member generates this companion file
- **CLAUDE.local.md bootstrap** — auto-wires context loading at session start; idempotent

No other artifact touches all four concerns (generation + persistence + session-loading + downstream file-map). The BMAD and CE members cover generation only; GSD covers one structural dimension (parallel mappers for tech/arch/quality/concerns).

## Existing-skill nesting

`system-context` already exists and is production-grade. This distillation is an **upgrade**, not a rebuild.

- **Existing mechanism**: Full 8-dimension generation, two-folder output, refresh detection, globs co-generation, session-loading bootstrap. Fires on "generate context", "orient me", "what does this app do", etc.
- **GSD parallel-mapper grain**: per-focus-area (tech / arch / quality / concerns). GSD's `map-codebase` spawns 4 parallel `gsd-codebase-mapper` agents each writing focused analysis documents. Currently `system-context` is a single-pass serial crawl.
- **CE phase grain**: per-scope (repo conventions / tech stack / implementation patterns). CE's `ce-repo-research-analyst` uses "configurable scoped phases."
- **Nesting rule**: For small/medium repos (current default), single-pass serial crawl is correct. For large repos (50+ source files, deep doc tree), add an optional `--parallel` mode that fans out 4 sub-agents for tech/arch/quality/concerns focus areas, then synthesizes. This is additive — default behaviour unchanged.

## Non-overlapping ideas folded in

- From `gsd:map-codebase` + `gsd-codebase-mapper`: **parallel fan-out across four codebase focus dimensions** — `complexity: medium | optional: true | prerequisite: repo has 50+ source files or extensive docs`. Serial crawl cannot fully cover a large codebase in a single pass; 4-way parallel coverage (tech stack, architecture, quality signals, concerns) produces richer CONTEXT.md for large repos at the cost of orchestration overhead.

- From `compound-engineering:agent:ce-repo-research-analyst`: **configurable phase scoping** — `complexity: low | optional: true | prerequisite: none`. CE's analyst accepts a scope parameter ("focus on test conventions only", "focus on API layer"). Adding optional scope hints to `system-context` invocation would let callers narrow what gets refreshed without full regeneration.

- From `bmad-method:skill:bmad-document-project`: **brownfield-first trigger phrase** — "document this project" currently routes to BMAD but arguably should route to `system-context`. The BMAD version produces a CLAUDE.md-level doc, not a CONTEXT.md comprehension document. `complexity: low | optional: false | prerequisite: none`. Add "document this project" as a trigger phrase and produce the same CONTEXT.md output — users who currently land on BMAD should land here instead.

- From `everything-claude-code:skill:codebase-onboarding`: **explicit "what NOT to summarize" guard** — ECC's onboarding skill flags certain file types (generated files, lock files, compiled assets) as explicitly excluded from comprehension. `complexity: low | optional: false | prerequisite: none`. Reduces noise in CONTEXT.md Core Abstractions section.
