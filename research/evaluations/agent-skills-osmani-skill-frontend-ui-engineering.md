# Eval: frontend-ui-engineering

**Intent**: Enforce production-quality UI standards — design-system adherence, accessibility (WCAG 2.1 AA), responsive design, and elimination of "AI aesthetic" patterns — when building or modifying user-facing interfaces.

## Scores
- **quality_score**: 5 — The "AI aesthetic anti-pattern table" (8 rows: purple/indigo everything, excessive gradients, etc.) is exceptional diagnostic content. Keyboard navigation, ARIA, focus management, empty/error/loading state patterns, and optimistic updates are all production-grade. The accessibility checklist redirect to `references/accessibility-checklist.md` shows good progressive-disclosure structure.
- **adoption_fit_final**: strong — Language-agnostic principles; examples are React/Tailwind but the concepts (composition over configuration, data/presentation separation, state hierarchy, accessibility rules) transfer broadly. Not stack-locked.
- **inspiration_value**: high — The "Avoid the AI Aesthetic" table naming eight specific AI-generated UI failure modes is uniquely valuable diagnostic content. The container/presentation split pattern with explicit loading/error/empty state requirements is reusable across any component architecture.
- **uniqueness_refined**: uncommon — "AI aesthetic" as a named, tableable anti-pattern category is not found elsewhere. The depth of accessibility coverage (keyboard, ARIA, focus management, meaningful empty states) goes further than any other UI skill in the corpus.
- **composability**: standalone — Operates as a set of constraints on any frontend implementation; referenced indirectly by `build` command.
- **description_craft_refined**: trigger — Description gives two trigger conditions ("when building or modifying user-facing interfaces") and names the outcome value ("look and feel production-quality rather than AI-generated").

## Mineable phrasing
> "UI that looks like it was built by a design-aware engineer at a top company — not like it was generated by an AI."

## Notes
The "AI aesthetic" anti-pattern table is the single most mineable piece of content in the stage 3 batch. Naming the eight failure modes (purple/indigo default, excessive gradients, rounded-2xl everything, generic hero sections, lorem ipsum copy, oversized padding, stock card grids, shadow-heavy design) gives agents a concrete checklist rather than a vague quality directive. The state management decision hierarchy (local → lifted → context → URL → server → global) is a clean prioritisation ladder that prevents over-engineering. The "Realistic placeholder content" anti-pattern note (lorem ipsum hides layout problems that real content reveals) is an undervalued diagnostic point.
