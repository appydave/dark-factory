The thesis
ONE FAILURE MODE · THREE CURES
All three projects independently hit the same wall: an orchestrator that poisons itself by accumulating every step's payloads — transcripts, intermediate values, wrong attempts — until it summarises, editorialises, or fabricates instead of passing data through. They prescribed three different cures.
YLO Blackboard
ISOLATE
Keep the LLM orchestrator, but take it out of the data path — workers are subagents with clean context that write the store directly and return one-line acks.
Bet: Architecture beats willpower.
POEM OS
DISCIPLINE
Keep the LLM orchestrator in the data path, and legislate good behaviour — ~600 lines of Oscar agent rules: no summarising, no augmentation, mechanical gate evaluation.
Bet: Rules can hold the line.
prompt.supportsignal
REMOVE
Delete the LLM from the orchestration seat entirely — compile the workflow to plain JavaScript that calls stateless provider.call(prompt, schema) per step.
Bet: No LLM orchestrator, no poisoning.

▤The Three Lenses

View each project as three separable things. They agree almost completely on lenses 1 + 2 — and diverge entirely on lens 3.
1 Prompt Engineering How are prompts authored, parameterised, versioned? ~90% aligned · all .hbs descendants of AWB Gen 3
YLO Blackboard
.hbs with bare {{markers}}
Output shape described in prose (“Return ONLY JSON: {…}”). No co-located schema (one exception). Informal file-level versioning. Lifted verbatim from AWB Gen 3.
loose contract
POEM OS
.hbs + co-located .json schema
Schema carries placeholders{type,required,example}, outputSchema, version. Statically type-checked: {{var}} / {{#each}} compatibility validated before any step runs.
validated contract
prompt.supportsignal
.hbs + .json (it IS a POEM install)
Each prompt pairs .hbs with .json carrying modelType (“quick”/“analysis”), maxTokens, temperature, outputSchema validated at runtime. Versioned + .old/ archives + git.
validated contract
2 Workflow Data Shape How is the multi-step pipeline declared? Same DAG vocabulary · POEM richer control flow
YLO Blackboard
JSON DAG
steps[]{ type, inputs[], stores, workerContract, humanGate, outputKeyMap }. Five step types. Pure dataflow dependency; gate exists, conditional-parallel not first-class.
lean
POEM OS
YAML DAG (richer)
action: llm / llm-parallel / llm-conditional-parallel, gate checks[] with onPass/onFail routing, skipIf / condition, active:false disable cascades. Materially richer branching.
rich control flow
prompt.supportsignal
YAML DAG · new-incident.yaml v1.4.0
Gate (invisible auto data-readiness) vs Checkpoint (visible human decision) categorically split. seed.mapping snake→camel, output.{drop,group} reshape. The production exemplar.
rich control flow
3 Architecture · Runner · Isolation What executes it? How is state held? How is context isolated? ⚡ The fork — they diverge entirely
YLO Blackboard
Orchestrator IS an LLM · isolation is BEHAVIOURAL
The Claude Code session is the conductor. Isolation enforced by subagent spawning + ack-only logs — provable by grep (no payload in log, no bytes in store).
state: append-only EAV JSONL · immutable history isolate
POEM OS
THREE runtimes · Oscar (LLM) failed
(1) render-only TS engine — no LLM call at all; (2) Oscar conversational agent — the LLM orchestrator that failed in field tests; (3) poem-executor CLI — compiled JS.
state: single mutable complete.json · snapshot discipline
prompt.supportsignal
Orchestrator is COMPILED JS · isolation STRUCTURAL & free
Compiler emits .compiled.js; run() calls provider.call(renderedPrompt, schema) per step. Each LLM touch is a stateless single-shot — no conversational context to poison.
state: flat in-memory store · saved once remove

⛓The Oscar Lineage

YLO'S “OSCAR” IS LITERALLY POEM'S OSCAR
  • YLO's “Oscar” — the naive just-run-it-in-main-context failure the blackboard pattern fixes — is literally POEM's Oscar agent.
  • POEM's docs never contain the words “context poisoning” — zero hits in the repo. That phrase is YLO's retro-label.
  • What POEM actually documented (Oscar.md, Rounds 1–3): completing workflows with fabricated data, condensing 800-word observations to 150, injecting restrictive qualifiers into predicate questions.
  • Same pathology, two vocabularies: YLO calls it “context poisoning”; POEM calls it “interpretation / fabrication.”
  • Governance scar: Oscar + Alex exist only in the client's .claude/commands/ — not in POEM core (poem-001 blocker, severity high). Field improvements have no sync path home.

⋈Where All Three Agree

Common ancestor
All three descend from AWB Gen 3
  • Same .hbs Handlebars templates with {{placeholder}} injection
  • Same five step types — transform / checkpoint / action / gate / elicit
  • Same declared inputs[] / outputs (stores) name-matched dataflow
  • Same output.{drop,group} final-projection reshape
  • Same “human gates forward decisions, not chatter” discipline
Agreement is total on lenses 1 + 2 (prompts + workflow shape). Divergence is total on lens 3 (architecture).

⇄Cross-Pollination — What Each Should Steal

YLO should steal from POEM
  • Co-located .json output schemas — validate worker writes mechanically instead of hoping the worker returns the right shape. YLO's single biggest gap.
  • A preflight / validate pass (validate-workflow.yaml) — catch type mismatches and dangling refs before any step runs.
  • A modelType alias layer (“analysis”/“quick” → concrete slug) — would have prevented the flux-schnell phantom-model bug (open loop #1).
POEM / supportsignal should steal from YLO
  • Append-only EAV audit store — every attempt, error, and human decision preserved. A compliance asset for the NDIS / safety-regulated domain, not just a nicety.
  • Resumability from the store — re-read, skip done steps, append a fresh attempt.
  • Structural isolation for the Oscar-as-LLM path — replace 600 lines of “don't summarise” rules with subagent isolation, where the rules become unnecessary.

§The Verdict

YLO is a working prototype of what POEM's Oscar was trying to be.
Lenses 1 + 2
prompts + workflow spec
easy
~90% shared already — both descend from AWB Gen 3. POEM YAML ↔ YLO JSON is mechanically convertible. They could share one schema.
Lens 3
runner / architecture
identity decision
Not a tweak — a choice of which POEM runtime survives: in-session conductor (YLO) vs external compiled CLI (poem-executor).
Recommendation: If POEM commits to in-session execution, adopt YLO's conductor as POEM's runtime and demote poem-executor to the headless / CI lane. The four probes have already de-risked it.
