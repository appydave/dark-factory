BState Model — Delete the Mutability Enum

In event sourcing, mutability isn't a storage property — it's a projection (roll-up) policy. Five write-time modes collapse into five read-time folds over one append-only log.
AWB mutability mode → projection fold
mutable→last-write-wins
immutable-after-first-write→first-write-wins
append-only→keep-all
replace-with-review→last-write-wins, gated by a review event
derived→computed fold over other keys
A relational-looking output record (titles, suggestedTitles as columns) is just a projection shape. "Mutating" a value = append an event + re-project. No mutation logic in the store.
The three separable layers
layer 1
Append-only log
one write model — YLO's blackboard substrate. Every fact immutable.
layer 2
Projection policies
the read-side fold — replaces the mutability MODES. Relational output is a projection shape.
layer 3
Write-time validators
optional guard rails as checks over the log (immutable key folds to >1 ok-event → raise). Where AWB's "shaped appropriately" lives.
A projection changes how you read — it can't reject a bad write. That's why layer 3 exists. But validators are rules over the log, not storage modes — the substrate stays dumb.
